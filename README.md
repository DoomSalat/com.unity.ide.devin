# com.unity.ide.devin

Unity Editor integration for [Devin IDE](https://devin.ai).

Registers Devin as an external script editor in Unity â€” auto-discovery, file opening with line navigation, and `.csproj` / `.sln` generation for IntelliSense via OmniSharp.

---

## Two variants

This repository provides two independent implementations on separate branches. Choose the one that fits your setup.

### `master` â€” Self-contained generation

Generates `.csproj` and `.sln` files from scratch using Unity's `CompilationPipeline` API. No other editor plugin required.

**Use when:** You have no Rider or Visual Studio installed, or you want full control over what gets generated.

**Trade-offs:**
- Independent of other editor plugins â€” works out of the box
- Configurable per package type: Embedded, Local, Git, Registry, Built-in, Player assemblies
- Requires .NET SDK 6.0+ on the machine (OmniSharp uses it to evaluate `.csproj` files)

**Installation:**
```json
"com.unity.ide.devin": "https://github.com/DoomSalat/com.unity.ide.devin.git"
```

---

### `via-reflection` â€” Delegate to installed editor *(faster)*

Delegates `.csproj` / `.sln` generation to an already-installed editor plugin â€” Rider or Visual Studio Tools â€” via Unity's `IExternalCodeEditor` API and internal reflection. Devin acts as a thin proxy that triggers the other editor's generator and then writes `omnisharp.json` and `Directory.Build.props` on top.

**Use when:** You already have `com.unity.ide.rider` or `com.unity.ide.visualstudio` in your project and want IntelliSense files generated by the same engine those tools use internally.

**Why it's faster:**
- Rider's and VS Tools' generators are heavily optimized and cache assembly data between runs
- Incremental sync (`SyncIfNeeded`) is handled by the delegate â€” only changed assemblies are re-evaluated
- VS Tools' internal `ProjectGenerator.Sync()` is called directly via reflection, bypassing the current-editor guard that would otherwise block it when Devin is selected

**Visual Studio discovery on Unity < 6000.5:**
`com.unity.ide.visualstudio` 2.0.27 has a bug on Unity versions below 6000.5: it resolves `vswhere.exe` to a wrong CWD-relative path, so VS discovery returns empty. `via-reflection` works around this automatically â€” when the plugin's own discovery fails, it finds `vswhere.exe` via `PackageInfo.resolvedPath` (always correct regardless of Unity version), runs it directly, and bootstraps the installation. The Preferences panel reflects this: if auto-discovery failed but the fallback succeeded, it shows "found via vswhere fallback" instead of a warning.

**Trade-offs:**
- Requires `com.unity.ide.rider` or `com.unity.ide.visualstudio` present in `Packages/manifest.json`
- Generated `.csproj` style depends on the delegate (Rider â†’ SDK-style, VS Tools â†’ Legacy or SDK depending on VS version â‰Ą 18)
- Uses internal reflection â€” may break if Unity changes editor plugin internals between versions

**Installation:**
```json
"com.unity.ide.devin": "https://github.com/DoomSalat/com.unity.ide.devin.git#via-reflection"
```

**GUI:** The Preferences panel shows a dropdown to pick which registered editor to delegate to, the installation status of the selected editor, and the live state of `.sln`, `omnisharp.json`, and `Directory.Build.props`.

## Common requirements

- Unity 2021.3+
- Devin IDE installed at:
  - **Windows:** `%LOCALAPPDATA%\Programs\Devin\Devin.exe`
  - **macOS:** `/Applications/Devin.app`
  - **Linux:** `/usr/bin/devin` or `/usr/local/bin/devin`

`master` additionally requires .NET SDK 6.0+ (OmniSharp uses it to load SDK-style `.csproj` files â€” Visual Studio is not required).

`via-reflection` delegates requirements to the chosen editor plugin (e.g. Rider installed locally, or Visual Studio installed for VS Tools).

## Installation (local package)

Use an **absolute path** to avoid Unity path-resolution issues:

```json
"com.unity.ide.devin": "file:D:/path/to/com.unity.ide.devin"
```

Relative paths (`file:../com.unity.ide.devin`) should also work per Unity docs (relative to project root), but absolute paths are more reliable in practice.

> **Note:** If the `package.json` contains a `_fingerprint` field, remove it before adding as a local package. Unity adds this field when caching git packages â€” it can prevent correct loading of a local reference.

## Setup

1. Open **Edit â†’ Preferences â†’ External Tools**
2. Set **External Script Editor** to **Devin**
3. *(master only)* Configure which package types to generate `.csproj` for
4. *(via-reflection only)* Select the delegate editor from the dropdown
5. Click **Regenerate project files**

OmniSharp will pick up the solution automatically via the generated `omnisharp.json`.

## OmniSharp notes

Both variants write two support files after every sync:

**`omnisharp.json`** â€” points OmniSharp at the correct `.sln`:
```json
{
  "msbuild": {
    "solution": "D:\\MyProject\\MyProject.sln",
    "loadProjectsOnDemand": false
  }
}
```

**`Directory.Build.props`** â€” prevents MSBuild from resolving .NET Framework targeting packs (required when Visual Studio is not installed):
```xml
<Project>
  <PropertyGroup>
    <SkipGetTargetFrameworkProperties>true</SkipGetTargetFrameworkProperties>
    <NoRestore>true</NoRestore>
  </PropertyGroup>
</Project>
```

`master` generates SDK-style projects (`<Project Sdk="Microsoft.NET.Sdk">` with `<TargetFramework>net471</TargetFramework>`). Assembly references not included in generation are resolved as `HintPath` entries pointing to `Library/ScriptAssemblies/`.

## Features

- **Auto-detects Devin** on Windows, macOS, and Linux
- **Opens files** at the correct line and column via `--goto`
- **Filters file types** â€” non-code assets (prefabs, materials, scenes) stay in Unity's own editors
- **`omnisharp.json`** auto-generated with the correct solution path
- **`Directory.Build.props`** auto-generated to keep MSBuild compatible with .NET Core SDK

## Troubleshooting

### OmniSharp crashes with .NET 10 â€” `NullReferenceException at GetBuildEnvironmentInfo()`

**Symptom:** OmniSharp log shows:
```
OmniSharp server started with .NET 10.0.x
[fail]: OmniSharp.WorkspaceInitializer â€” System.NullReferenceException
   at OmniSharp.MSBuild.MSBuildHelpers.GetBuildEnvironmentInfo()
```
No projects load. IntelliSense is completely absent.

**Cause:** OmniSharp 1.39.x is compiled for `net6.0` but .NET automatically picks the highest available compatible runtime â€” .NET 10 in this case. The .NET 10 runtime changed internal MSBuild APIs that OmniSharp depends on, causing a crash at initialization.

**Fix:** Pin OmniSharp to use the .NET 9 runtime by editing its `runtimeconfig.json`. Find the file next to `OmniSharp.dll` in your C# extension folder, e.g.:
```
%USERPROFILE%\.devin\extensions\muhammad-sammy.csharp-<version>\
    .omnisharp\1.39.14-net6.0\OmniSharp.runtimeconfig.json
```
Change `rollForward` and `version`:
```json
{
  "runtimeOptions": {
    "tfm": "net6.0",
    "rollForward": "LatestMinor",
    "framework": {
      "name": "Microsoft.NETCore.App",
      "version": "9.0.0"
    }
  }
}
```
`LatestMinor` with `9.0.0` uses the latest .NET 9.x patch and will not roll forward to .NET 10.

Restart OmniSharp after the change (`Ctrl+Shift+P â†’ Restart OmniSharp`). The log should now say `OmniSharp server started with .NET 9.0.x`.

> **Note:** This file may be overwritten when the extension updates. Re-apply the fix after each update.

### Unity stuck on "Resolving packages" at startup

**Symptom:** Unity hangs indefinitely on the "Resolving packages" progress bar when the package is referenced via git URL.

**Cause:** A stale git-cache entry in `Library/PackageCache` and/or `Packages/packages-lock.json` still references the old git URL while the package has been removed or changed.

**Fix:**
1. Close Unity
2. Delete the cached package folder:
   ```
   Library/PackageCache/com.unity.ide.devin@<hash>/
   ```
3. Remove the entry from `Packages/packages-lock.json`:
   ```json
   // delete this block:
   "com.unity.ide.devin": {
     "version": "https://github.com/...",
     ...
   }
   ```
4. Make sure `Packages/manifest.json` does not contain a `com.unity.ide.devin` git URL entry (or replace it with the local path you want)
5. Open Unity â€” package resolution will complete normally

### Local package not found â€” `package.json cannot be found`

**Symptom:** Unity shows an error about `package.json` not being found even though the folder exists.

**Causes and fixes:**

**1. Wrong path format.** Use an absolute path to eliminate ambiguity:
```json
"com.unity.ide.devin": "file:D:/GameDev/UnityEngine/Projects/com.unity.ide.devin"
```
Relative paths are resolved from the project root (the folder containing `Assets/`), but absolute paths are safer.

**2. `_fingerprint` field in `package.json`.** Unity adds this field when caching a git package in `Library/PackageCache`. If you cloned the repo while it was previously added via git URL, the cached `package.json` may contain:
```json
"_fingerprint": "..."
```
Remove this field from `package.json` in the local repository. It is only used internally by Unity's git package cache and breaks local package loading.

### via-reflection: Visual Studio not found â€” Preferences shows warning after Regenerate

**Symptom:** After clicking **Regenerate project files**, the Preferences panel still shows "Visual Studio installations were not auto-discovered."

**Cause:** `com.unity.ide.visualstudio` 2.0.27 has a bug on Unity < 6000.5 where `vswhere.exe` is resolved to a nonexistent CWD-relative path (`Packages/com.unity.ide.visualstudio/Editor/VSWhere/vswhere.exe`) instead of the actual package cache path. VS discovery returns an empty dictionary.

The `via-reflection` branch includes an automatic fallback: on the first **Regenerate** click it runs `vswhere.exe` directly (found via `PackageInfo.resolvedPath`) and bootstraps the VS installation. After that the Preferences panel should update to say "found via vswhere fallback".

**If the fallback also fails:**
- Confirm Visual Studio (Community, Professional, or Enterprise) is installed
- Run `vswhere.exe -latest -property installationPath` from a terminal â€” it should output an install path
- If vswhere returns nothing, VS workload "Desktop development with C++" may be missing; open Visual Studio Installer and add it

As an alternative, downgrade `com.unity.ide.visualstudio` to `2.0.22` in `Packages/manifest.json` â€” that version does not have the path bug:
```json
"com.unity.ide.visualstudio": "2.0.22"
```
